### 问题清单

#### 🔴 严重问题
**问题1：LangChain 工具函数缺少描述，导致包导入即崩溃**
- 位置：`backend/src/slothbearflow_backend/tools/time_tool.py:8`、`backend/src/slothbearflow_backend/tools/weather_tool.py:6`、`backend/src/slothbearflow_backend/tools/session_tool.py:11`、`backend/src/slothbearflow_backend/tools/rag_tool.py:44`
- 问题：`@tool` 装饰的函数没有 docstring，也没有显式传入 description。当前依赖版本下，LangChain 在创建 `StructuredTool` 时抛出 `ValueError: Function must have a docstring if description not provided.`
- 风险：任何导入 `backend.src.slothbearflow_backend` 的路径都会触发工具注册，API、测试和本地启动都会失败。当前 `pytest` 的 22 个失败和 1 个错误基本都被这个问题阻断。
- 建议：给每个 `@tool` 函数补充清晰 docstring，或改为 `@tool("name", description="...")`；修复后重新跑全量 smoke tests。

**问题2：`.env` 与运行日志已被 Git 跟踪**
- 位置：`.env`、`logs/app.log`、`logs/error.log`
- 问题：`.gitignore` 虽然包含 `.env`、`*.log`、`logs/`，但这些文件已经在索引中，后续测试/启动会持续产生 diff。`.env` 目前未见已设置 API key，但仍包含运行配置和 PostgreSQL DSN，后续很容易误提交敏感信息。
- 风险：配置泄露、日志泄露、日常开发 diff 污染；如果真实 token 被写入 `.env`，普通 `.gitignore` 已无法阻止提交。
- 建议：确认无敏感历史后执行 `git rm --cached .env logs/app.log logs/error.log`，保留 `backend/.env.example` 作为模板；如曾提交真实密钥，需轮换密钥并清理历史。

#### 🟡 建议修改
**问题3：PostgreSQL 会话恢复 SQL 的外层排序引用了未投影的 `id`**
- 位置：`backend/src/slothbearflow_backend/persistence/postgres.py:300`
- 问题：子查询只选择了 `user_message, assistant_message`，外层却 `ORDER BY id ASC`。启用 `POSTGRES_RESTORE_ON_REDIS_MISS=true` 后，真实数据库查询会失败并回退为空会话。
- 建议：子查询中同时选出 `id`，外层排序后再在 Python 中只消费消息字段；或使用 CTE 明确保留排序字段。

**问题4：RAG 引用信息通过 `ContextVar` 写入，但 Agent 调用运行在 worker thread**
- 位置：`backend/src/slothbearflow_backend/main.py:504`、`backend/src/slothbearflow_backend/tools/rag_tool.py:34`
- 问题：`/chat` 使用 `asyncio.to_thread(executor.invoke, ...)`，而 `search_knowledge` 在工具执行线程里设置 `ContextVar`。`to_thread` 会复制上下文进入线程，但线程内对 `ContextVar` 的写入不会传播回请求协程，随后 `get_last_rag_sources()` / `get_last_rag_citations()` 可能读不到真实引用。
- 风险：RAG 实际检索成功，但 API 返回的 `source`、`citations`、`tools_used` 为空或不准确。
- 建议：不要用跨线程 `ContextVar` 作为结果通道；改为工具返回结构化 artifact、把引用挂到 request-scoped collector，或让 executor 显式返回工具调用元数据。

**问题5：配置项存在但没有生效**
- 位置：`backend/src/slothbearflow_backend/config.py:114`、`backend/src/slothbearflow_backend/agent/react_runtime.py:70`
- 问题：`REACT_TOOL_TIMEOUT_SEC` 已暴露在配置中，但 `ExplicitReActRuntime._invoke_tool()` 没有使用超时控制；工具调用卡住时会阻塞整次 Agent 调用。
- 建议：在显式 ReAct runtime 中对工具执行加超时，或移除暂不支持的配置以免误导。
